# Preprocessing And BCC Performance Shape

This document records the production execution model for TALUS nonce
preprocessing, CEF, CarryCompare, and BCC certification.

The rule is the same as for DKG:

```text
Do not implement production preprocessing as independent scalar work per
coefficient, per token, or per signer.
```

Current non-goals:

- Do not restore public exact `A*y`, `A*nonce`, or Feldman-style
  `Phi = A*secret` commitments.
- Do not add post-challenge reveal-on-failure to production.
- Do not build a second preprocessing algorithm. Replace internals behind the
  existing session/verifier/token boundaries.

Production preprocessing must be vectorized and batched across:

- token batches
- all coefficients in each token
- all signers in the signing set
- nonce-share generation
- masked-broadcast commit/open
- masked-broadcast consistency certification
- CEF high/low arithmetic
- CarryCompare comparison layers
- BCC boundary checks
- all openings in the same phase
- all bitness/range/zero checks in the same phase
- durable logs and transport payloads

If preprocessing takes days, the implementation is wrong. It is allowed to be
heavier than online signing, but it must amortize work across token batches and
vector lanes. The target execution shape is seconds to tens of seconds on LAN
for practical batch sizes, and tens of seconds to a few minutes on WAN depending
on RTT, party count, durable logging, and batch size.

## Token Batch Model

Production signing consumes BCC-certified tokens. Therefore preprocessing should
produce tokens in batches:

```text
TokenBatch<P> {
  suite
  signing_set
  epoch
  session_id
  token_count = K
  coefficient_lanes = P::K * 256
}
```

The vectorized lane shape is:

```text
lane = (token_index, coefficient_index)
```

or chunked as:

```text
chunk = bounded set of (token_index, coefficient_index) lanes
```

Chunking is allowed for memory and MTU limits. Falling back to one MPC/VSS
execution per coefficient is not.

## Distributed Nonce Generation

Nonce shares must be generated by production IT-VSS/IT-MPC, not by caller
provided `y_shares`.

Production nonce generation should share whole token vectors or bounded chunks:

```text
one vector sharing per dealer per token batch
```

or:

```text
one vector sharing per dealer per bounded token-batch chunk
```

It must not use:

```text
one VSS instance per coefficient
one VSS instance per signer/coefficient pair
one scalar complaint path per coefficient
```

The VSS security logic is unchanged by batching. Each vector IC tag
authenticates an `F_q^M` vector:

```text
c_vec = beta_vec + b * y_vec
```

where `b` is hidden from the holder and receiver-private retained tags are never
public. Independent repeated tags provide the configured soundness margin.

## Masked Broadcast Commit/Open

Each signer computes its local contribution to `A*y_i`, decomposes it into
unsigned high/low parts for CEF, masks the high and low values, commits, and
opens through reliable broadcast.

Production must batch:

- high masks for all lanes
- low masks for all lanes
- masked high values
- masked low values
- commitments for all lanes in a token batch or chunk
- reliable-broadcast delivery for each signer
- consistency checks for all lanes in a phase

The public transcript may contain only committed/opened masked values and public
certificates. It must not contain nonce shares, unmasked `A*y_i`, unmasked low
bits, mask shares, private consistency witnesses, or public exact `A*secret`
commitments.

## CEF And CarryCompare

CEF reconstruction and CarryCompare must run as vector circuits over all lanes.

For each lane, the protocol computes:

```text
B = sum_i btilde_i
t = B mod alpha
R = sum_i rho_i
kappa = [R > t]
delta = [R < t - gamma2 + kappa*alpha]
w1 = (sum_Htilde + floor(B/alpha) - kappa + delta) mod m
```

The production circuit must compute these as vector operations:

- vector sum of masked lows
- vector public `t = B mod alpha`
- vector secret/public comparison for `kappa`
- vector correction comparison for `delta`
- vector CEF high reconstruction
- batched checks before token certification

The implementation must not name the carry bit `c`; use `kappa`,
`rho_carry`, or `mask_carry` to avoid collision with the ML-DSA challenge
polynomial.

## BCC Certification

BCC certification is the admission gate for strict production tokens.

For each token, BCC checks that the aggregate nonce commitment is far enough
from `HighBits` boundaries:

```text
|LowBits(A*y)| < gamma2 - beta
```

Production should implement BCC as a private vector predicate over all
coefficient lanes:

- compute or certify the required high/low decomposition privately
- compare boundary distance for every lane
- aggregate per-lane bits into a per-token admission bit
- certify only tokens whose admission bit is true
- discard failed tokens before challenge

Failed BCC is pre-challenge and retryable. It must not reveal nonce shares,
boundary distances, low bits, per-coefficient failure positions, or masks.

## Precomputation

Preprocessing should precompute and certify one-time material before it is
needed by signing:

- nonce VSS vector sharings
- CEF high/low masks
- CarryCompare random bits and comparison masks
- BCC comparison masks
- certified Beaver/IT-MPC multiplication resources, if the backend requires
  preprocessing
- durable token ids and one-time-use records

All such material is one-time use. Reuse after crash or rollback is forbidden.

Precomputed material must be transcript-bound to:

```text
suite
epoch
preprocessing session id
signing set
party set hash
token batch id
token index or chunk id
coefficient lane range
```

## Security Of Batching

Batching changes scheduling and payload shape, not the algebraic security
statement.

The implementation remains secure only if:

- every lane is transcript-bound with token/chunk/coefficient identity
- randomness is independent where the protocol requires independence
- retained IC tags remain receiver-private
- audited tags are discarded
- bitness/range/zero checks are performed before dependent openings
- openings are checked and authenticated
- public branching does not depend on unchecked private values
- failed token details are not opened
- token ids are consumed durably before online response work

Do not reuse one mask, IC tag, random bit, or Beaver triple across lanes unless
the protocol explicitly proves that reuse safe. The default rule is independent
randomness per lane and batched verification of those lanes.

## Counters

Production preprocessing must expose counters for:

```text
token batches
tokens requested
tokens certified
tokens rejected before challenge
vector lanes
chunks
IT-VSS vector sharings
IT-VSS private deliveries
IC tags
consistency rounds
MPC multiplication layers
MPC gates
open_many phases
assert_zero/assert_bit phases
private messages
broadcast messages
wire bytes
durable log bytes
wall-clock time
```

These counters are release gates. Message and round counts must scale with
token batches, chunks, and circuit depth, not with scalar coefficient loops.
